0 · Назначение и границы

Документ фиксирует то, что должно быть одинаковым на всех платформах: доменную модель данных, состояния системы, состав и назначение экранов, поведение диаграммы, правила контроля допусков, логику сценария проверки и визуальный язык. Реализация (Qt-виджеты, веб, мобайл) остаётся за командами платформ.

1 · Принципы
1
Источник истины — контракт данных, не код платформы
Формат канала, кадра и сценария — платформо-независимы. Qt-ядро и веб-клиент — это разные «головы» к одному телу данных.
2
Сырые отсчёты АЦП — первичны
График всегда отображает сырой код (0–1023) столбцами. Перевод в физ. величину — необязательный слой поверх, появится позже.
3
Цвет несёт смысл, а не украшает
Столбцы одного нейтрального цвета; акцент (жёлтый/красный) появляется только при выходе кода за допуск.
4
Переносим намерения, а не пиксели
Для Qt важна иерархия, поведение и ритм обновления. Для веб/мобайл прототип переносится буквальнее.
2 · Доменная модель данных

3 · Состояния системы

Три независимые оси состояния. Веб/мобайл-клиент добавляет ось связи, которой нет у локального спайк-пульта — её нужно проектировать заранее.

Запись
не пишется ⏺ идёт запись
Связь ⟂
приём отставание / потери кадров нет связи / реконнект
Только для сетевых клиентов (веб/мобайл). У локального Qt-пульта в спайке этой оси нет — но дизайн обязан её предусмотреть.
4 · Экраны

Многооконность как переключение основной области. На десктопе допустимы отдельные окна ОС; в браузере — экраны/роуты в одной вкладке.

Сбор
Живая столбчатая диаграмма всех каналов + крупный отсчёт выбранного. Компоновки A (списки + диаграмма + контроль допусков), B (диаграмма + таблица точных значений), C (карточки по категориям).
Детально
Один канал: крупный код над диаграммой, история во времени столбцами с линиями допуска, имя/адрес внизу, статистика мин/сред/макс/размах.
Конфигуратор
Список файлов конфигурации (.txt), кодировка, предпросмотр каналов, применение.
База
Библиотека параметров: дерево категорий + редактор метаданных и шкала допуска.
Сценарий ◇
Модальный мастер поверх любого экрана. Развилка файл / вручную, шаги команд и проверок, автопрогон с вердиктом.
5 · Поведение диаграммы
  • Высота столбца = текущий сырой код АЦП (шкала 0–1023). Только столбцы — не линии.
  • Каналы сгруппированы по категориям; подпись категории — под группой столбцов.
  • Наведение — подсветка столбца; одиночный выбор — синхронно в списке, диаграмме и панели контроля; двойной клик / «Детально» — открытие детального экрана.
  • Над диаграммой — точное значение выбранного канала крупным моноширинным; имя/адрес — снизу.

7 · Сценарий проверки

Последовательность шагов двух видов, выполняемая автопрогоном. Развилка на входе: загрузить файл (.scn) либо собрать вручную.

Шаг · Команда
Воздействие оператора/КПА: «Подать питание 27 В», «Включить наддув». Помечается выполненным.
Шаг · Проверка
Канал + ожидаемый допуск. Результат ОК, если code ∈ [lo, hi], иначе НЕ ОК.
Автопрогон: идёт по шагам → отметка ОК/НЕ ОК → счётчики и прогресс → итоговый вердикт ОКНЕ ОК

9 · Адаптация по платформам
Платформа Особенности
Десктоп (Qt)Плотная сетка, наведение мышью, узкие хит-зоны допустимы. Отдельные окна ОС. Спайк — без оси связи.
БраузерЭкраны = роуты в одной вкладке. Прототип переносится почти буквально. Появляется ось связи.
МобайлНет наведения. Хит-зоны ≥ 44px. Вероятна 4-я, вертикальная компоновка. Диаграмма — свайп/зум.
10 · Глоссарий
МТВ — бортовое время (метка времени кадра)
АЦП — код 0–1023, первичная величина графика
ЯТП — набор каналов телеметрии
БСИ — блок сбора информации (категории)
спайк — прямая связь с логикой, не по сети
допуск — [lo, hi] вокруг номинала
