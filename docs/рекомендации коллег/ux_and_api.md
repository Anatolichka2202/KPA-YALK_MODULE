# Архитектура ядра, API и пользовательские сценарии

## 

`ChannelRegistry` — это **не виджет и не модель UI**, а инфраструктурный
компонент внутри ядра. Решает одновременно три разные проблемы:

### Проблема А — гонка данных

```
Поток декодера (вызывается из decoderLoop):
    for (auto& ch : channels_)         // <-- читает каждые ~31 мс
        process(ch);

Поток UI (когда оператор сменил конфиг):
    channels_ = new_channels;          // <-- пишет
```

Если эти два события пересекутся:
- `std::vector` при переприсваивании освобождает старый буфер
- Декодерный поток держит итератор на освобождённую память
- Сегфолт или хуже — тихая порча значений

**Простой мьютекс не решает**:

```cpp
// Вариант 1 — держать мьютекс весь цикл декодера
std::lock_guard lock(channels_mutex_);
for (auto& ch : channels_) process(ch);   // 100-300 µs под локом
// → UI замирает на эти 300 µs каждые 31 мс, заметно "лагает"
```

```cpp
// Вариант 2 — мьютекс на каждый канал
for (size_t i = 0; i < channels_.size(); ++i) {
    std::lock_guard lock(channels_mutex_);  // 64 lock/unlock на группу
    process(channels_[i]);
}
// → overhead огромный, плюс размер vector тоже надо защищать
```

**Решение через snapshot:**

```cpp
auto snap = registry_.get();          // ОДИН lock — копия shared_ptr (O(1))
for (auto& entry : snap->entries)     // итерация без блокировок
    process(entry);
// snap живёт пока есть shared_ptr → старый snap не разрушится
// даже если UI поставил новый
```

UI делает `registry_.update(...)` — старый snapshot продолжает жить пока
декодер не отпустит его (закончит обработку группы). Это классический
**RCU-паттерн** (Read-Copy-Update) через `shared_ptr`.

### Проблема Б — индексация в DataPool

Это был баг в исходном коде:

```cpp
// БЫЛО:
for (size_t i = 0; i < channels_.size(); ++i) {
    switch (channels_[i].adressType) {
    case ANALOG: data_pool_->setAnalog(i, ...);   // <-- БАГ
    case CONTACT: data_pool_->setContact(i, ...); // <-- БАГ
    }
}
```

`DataPool` хранит:
- `analog_[]`  — только аналоговые
- `contact_[]` — только контактные
- `fast_[]`    — только быстрые

Это **разные массивы** с per-type индексами от 0.
При смешанной конфигурации `[analog, contact, analog]` код вызывает:
- `setAnalog(0, ...)`   ← правильно
- `setContact(1, ...)`  ← неправильно, должно быть `setContact(0, ...)`
- `setAnalog(2, ...)`   ← неправильно, должно быть `setAnalog(1, ...)` или выход за границы

`ChannelRegistry::Entry` хранит **уже посчитанный** `poolIndex` —
тот самый правильный per-type индекс. Считается один раз при `update()`.

### Проблема В — связь канал ↔ метаданные

В `onDecoderGroup` мы видим `ChannelDesc`, но имя/категория параметра нужны
в разных местах:
- При записи в TLM
- При отдаче значения наружу через `getValue`
- При логировании "канал такой-то вышел за пределы"

`Entry` объединяет:
- `desc`       — что декодировать (wordIndex, groups, cycles)
- `poolIndex`  — куда положить значение
- `name`       — для оператора
- `category`   — для группировки в UI

### Альтернативы которые рассматривались

| Подход | Проблема |
|---|---|
| Мьютекс на `channels_` | Тормозит UI или декодер |
| `std::atomic<vector*>` + manual free | Сложно отследить когда освобождать |
| Lock-free queue команд | Усложнение без выгоды для нашего use-case |
| Двойная буферизация (active/pending) | Тот же RCU, но менее идиоматично |
| `std::atomic<shared_ptr<T>>` (C++20) | Требует C++20, который в Qt6 опционален |

`shared_ptr<const Snapshot>` + `std::mutex` для замены указателя —
**самый простой и идиоматичный** способ для C++17.

### Можно ли без него?

Можно. Если ты в `setChannelsFromPairs` берёшь `std::lock_guard` на тот же
мьютекс что и декодер при чтении — это работает, но:
- Каждый `getAnalog(i)` под локом
- Каждый `onDecoderGroup` под локом весь его цикл
- Размер блокировок неудобно отслеживать

Если у тебя так и сделано — расскажи подробнее, может работает приемлемо.
Но я бы рекомендовал snapshot-подход в любом случае.

## Часть 2. Какой функционал должно поддерживать ядро

Прежде чем "запирать" API, давай зафиксируем что в нём должно быть.
Принцип: **минимально, но достаточно**. То что нельзя реализовать снаружи
через текущие методы — должно быть в API. Остальное — нет.

### 2.1 Что должно быть в ядре

#### Источник данных
```cpp
void setDeviceE2010(int channel, double rate_khz);
void setDevicePlayback(const std::string& tlm_path);    // воспроизведение
void setDeviceNone();                                    // только feedRawData
```

#### Жизненный цикл сбора
```cpp
void start();
void stop();
void pause();              // ← новое: заморозить значения без stop
void resume();
bool isRunning() const;
bool isPaused() const;
```

#### Каналы — атомарные операции
```cpp
// Полная замена набора (горячая)
void setChannels(const std::vector<ChannelSpec>& specs);

// Чтение текущей конфигурации
std::vector<ChannelInfo> getChannels() const;

// Получение по адресу — не зависит от индекса
std::optional<double> getValue(const std::string& address) const;

// Снимок всех значений за одно обращение
struct Snapshot {
    uint32_t mtv_seconds;
    std::vector<ChannelValue> values;   // адрес + значение + valid
    Stats stats;
};
Snapshot getSnapshot() const;
```

`ChannelSpec`, `ChannelInfo`, `ChannelValue` — POD-структуры в публичном
заголовке, не зависят от `ChannelDesc` (тот — приватная деталь).

#### Уведомления о новых данных
```cpp
// Pull: блокирующее ожидание (уже есть)
bool waitForData(std::chrono::milliseconds timeout);

// Push: колбэк (опционально, для серверной части)
using DataCallback = std::function<void(const Snapshot&)>;
void setDataCallback(DataCallback cb);  // вызывается из декодерного потока
```

#### Статистика
```cpp
struct Stats {
    int phrase_error_percent;
    int group_error_percent;
    uint64_t frames_processed;
    uint64_t bytes_processed;
    double mb_per_second;
};
Stats getStats() const;
```

#### TLM-запись
```cpp
void startRecording(const std::string& path);
void stopRecording();
bool isRecording() const;
size_t recordedBytes() const;
```

#### Бортовое время
```cpp
uint32_t getCurrentTimeSeconds() const;
```

#### Управление сырыми данными (для тестов)
```cpp
void feedRawData(const int16_t* samples, size_t count);
```

### 2.2 Что НЕ должно быть в ядре

| Функция | Куда | Почему |
|---|---|---|
| Парсинг текстовых конфигов | UI / отдельный модуль | Файловые операции, кодировки — это слой выше ядра |
| База данных параметров | UI | Это конфиг для пользователя, не для декодера |
| Правила (пороги, тревоги) | UI / RuleEngine | Бизнес-логика, меняется чаще ядра |
| Графики, экспорт в CSV/PDF | UI | Презентационный слой |
| TCP/WebSocket-сервер | server/ модуль | Транспорт, не относится к декодированию |
| Lua-скриптинг | Отдельный модуль | Опционально, не нужно всем потребителям |
| Кодировки файлов | UI | См. парсинг конфигов |

### 2.3 Расширения которые стоит обдумать сейчас

Это не обязательно реализовывать сразу, но **API должен их допускать**:

#### A. История значений (для графиков)
Декодер уже сейчас выкидывает значения — если оператор хочет график за
последние 60 секунд, ему нужно самому собирать историю.

**Варианты:**
1. Хранить ring-buffer в ядре (`getHistory(address, seconds)`)
2. Hardcoded TLM-запись + чтение
3. Колбэк `setDataCallback` — пусть UI собирает сам

Я бы выбрал **(3)**: ядро отдаёт каждую группу, кто хочет — собирает.
Это проще и не раздувает ядро.

#### B. Подписка на конкретные каналы
Сейчас `onDecoderGroup` обрабатывает все каналы. Если оператор смотрит
один параметр на графике, остальные 100 каналов всё равно обрабатываются.

**Решение:** оставить как есть. Декодирование одного канала — это
несколько операций над `uint16_t`, не оптимизация на которой стоит
тратить силы.

#### C. Несколько устройств одновременно
Сейчас один `device_`. Если в будущем понадобится LimeSDR + E2010
одновременно — API сейчас этого не даёт.

**Вопрос к тебе:** это реалистичный сценарий или нет? Если нет —
закрепляем "одно устройство", и проблема не существует.

#### D. Многопоточные читатели
Если несколько потоков (например серверный обработчик клиентов) делают
`getSnapshot()` одновременно — это сейчас работает (мьютекс), но снапшоты
будут разные между ними. Норм для нашего случая.

#### E. Триггеры
> "вызови меня когда канал X вышел за пределы [min, max]"

Это можно реализовать снаружи через `setDataCallback` + RuleEngine.
В ядре делать НЕ нужно.

## Часть 3. Пользовательские сценарии

### Сценарий 1 — Первый запуск, новый оператор

```
1. Открывает OrbitaDesktop.exe
   → Видит главное окно: пустой график, кнопка "АЦП не подключено"

2. Меню: Устройство → Подключить E20-10
   → Диалог: канал [0..3], частота [10000 кГц по умолчанию]
   → Жмёт OK
   → Кнопка меняется на "АЦП: канал 0, 10 МГц"

3. Жмёт "Старт"
   → Индикатор внизу: "Сбор активен, МТВ 00:00:00"
   → График пустой (каналов нет)
   → Декодер крутится, статистика обновляется

4. Левая панель: "Параметры" — дерево по категориям из БД
   → Раскрывает "БСИ МКА-1 аналоговые"
   → Чекбоксы у нескольких параметров
   → Кнопка "Применить выбранные" внизу панели

5. Жмёт "Применить"
   → orbita->setChannels(...) — горячая замена
   → График мгновенно начинает рисовать значения
   → Внизу появляется таблица с текущими значениями
```

**Что нужно в API ядра:**
- `setChannels` без необходимости перед этим вызывать stop
- `getStats` для индикатора активности
- `getSnapshot` для таблицы значений

### Сценарий 2 — Работа с файлом конфигурации

```
1. Сбор уже идёт, каналы выбраны вручную
2. Меню: Конфигурация → Загрузить из файла
   → Список .txt файлов из data/configs/
   → Превью: количество адресов, какие из них есть в БД
3. Выбирает "МКИО_1 Потоки 1-3.txt"
   → Жмёт "Применить"
   → Файл читается через encoding_utils
   → Адреса нормализуются, ищутся в БД, формируются пары
   → orbita->setChannels(pairs)
   → График перерисовывается под новый набор
4. В заголовке окна: "Конфиг: МКИО_1 Потоки 1-3"
```

**Что нужно в API ядра:** только `setChannels` с парами. Чтение файла,
нормализация, поиск в БД — это всё UI.

### Сценарий 3 — Запись и воспроизведение TLM

```
Запись:
1. Сбор идёт, конфиг загружен
2. Кнопка "REC" (красная)
   → Диалог: куда сохранять, имя файла (по умолчанию orbita_2025-01-15_143022.tlm)
   → Жмёт Сохранить
   → Внизу: "REC ● 00:01:23 / 47 MB"
3. Жмёт "Stop REC"
   → Сообщение: "Записано 47 МБ в orbita_*.tlm"

Воспроизведение (другой день):
1. Меню: Файл → Открыть TLM
   → Выбирает orbita_2025-01-15_143022.tlm
2. Программа сама:
   - Останавливает текущее устройство
   - Переключается на setDevicePlayback(path)
   - Если в файле есть метаданные о каналах — подгружает их
3. Запускает воспроизведение
   → Тот же UI, те же графики, но данные из файла
   → Можно поставить на паузу, перемотать
```

**Что нужно в API ядра:**
- `setDevicePlayback(path)` — новый метод
- `pause()` / `resume()` — пауза без потери позиции в файле
- (опционально) `seek(position)` — перемотка

### Сценарий 4 — Тревоги и нарушения

```
1. Параметр "Напряжение БС" вышел за [3.0 .. 4.5] В
   → Строка в таблице подсвечивается красным
   → В журнале (Tools → Журнал событий) запись:
       "14:23:05 ALARM Напряжение БС = 4.71 V (max 4.50)"
   → (опционально) звуковой сигнал
2. Параметр вернулся в норму
   → Подсветка снимается
   → Запись: "14:23:18 OK Напряжение БС = 4.31 V"
3. Оператор открывает Tools → Журнал
   → Список всех событий с фильтром по дате/каналу/типу
```

**Что нужно в API ядра:** ничего нового! `setDataCallback` + UI-сторонний
`RuleEngine` который сравнивает значения с порогами. Журнал — тоже UI.

### Сценарий 5 — Изменение конфига на лету без остановки

```
1. Сбор идёт, конфиг "МКА-1" активен
2. Оператор замечает что в "МКА-1" не хватает каналов которые он хочет смотреть
3. Открывает левую панель параметров
4. Добавляет ещё 3 параметра галочками
5. Жмёт "Применить"
   → orbita->setChannels(старые + новые)
   → Графики продолжают работать без рывка
   → Новые каналы появляются в таблице
6. МТВ ни на секунду не сбрасывался
```

**Это и есть главный аргумент за ChannelRegistry.** Если бы `setChannels`
останавливал/перезапускал поток — был бы заметный пропуск данных и
сброс МТВ. С атомарной заменой через snapshot — нет.

### Сценарий 6 — Несколько окон одного потока (продвинутый)

```
Один сбор → разные представления:
- Главное окно: общий график
- Окно "Аналоговые БСИ": только эта категория, таблица
- Окно "Контактные": бинарные индикаторы
- Окно "Журнал": последние 100 событий
```

**Что нужно в API ядра:** ничего особенного. Главное — `getSnapshot()`
безопасен для вызова из любого потока, и UI-окна (все в Qt-потоке)
просто опрашивают по таймеру.

## Часть 4. UX-принципы для оператора

Оператор — не разработчик. У него:
- Может быть слабое зрение (диспетчерская смена 12 часов)
- Стресс при тревогах — UI должен быть однозначным
- Не должен бояться "сломать" кнопкой

### Правила

1. **Не более 3 кликов до любой важной операции.**
   Старт сбора, выбор конфига, начало записи — каждая в один-два клика.

2. **Состояние всегда видно.**
   Где-то на экране в каждый момент должно быть видно:
   - Сбор идёт / стоит / пауза
   - Активный конфиг
   - МТВ
   - REC если идёт запись
   - % ошибок декодирования (если > 0 — жёлтое, > 5% — красное)

3. **Отмена опасных действий.**
   Перед `stop()` — подтверждение если идёт запись.
   Перед заменой конфига — превью что изменится.

4. **Цветовая семантика однозначная:**
   - Зелёный = норма / работает
   - Жёлтый = внимание (потеря фразы, низкое качество)
   - Красный = тревога / нарушение порога
   - Серый = неактивно / нет данных

5. **Никаких модальных диалогов во время сбора.**
   Тревога не должна перекрывать график. Это панель внизу или док.

6. **Большие шрифты для значений.**
   Текущее значение параметра — 18-24px. Не 11.

## Часть 5. Минимальные изменения в текущем коде для соответствия

Чтобы API ядра соответствовал сценариям выше, нужно добавить:

1. **`pause()` / `resume()` / `isPaused()`** — для воспроизведения и
   "заморозки" значений при анализе.

2. **`getSnapshot()` атомарный** — сейчас оператор делает 64 вызова
   `getAnalog(i)` без гарантии что значения из одной группы. Это
   приемлемо для графика, но не для отчёта.

3. **`setDataCallback(cb)`** — для серверной части и RuleEngine.

4. **`getChannels()` const** — чтобы UI мог отрисовать что сейчас активно.

5. **`getValueByAddress(addr)`** — поиск по адресу, не по индексу.
   Удобно когда оператор смотрит "тот же параметр" в разных конфигах.

6. **`setDevicePlayback(path)`** — режим воспроизведения файла.

Все остальные методы у тебя уже есть.
