## Основные принципы проектирования (запреты и правила)
### 1. Никакой GUI-кода внутри статической библиотеки – вся визуализация только в desktop/. 
на подумать: использование сигнальной ситсемы qt? 

### 2. Чёткое разделение на публичное и приватное – в include/ только то, что разрешено использовать внешним приложениям. Всё остальное – src/, закрыто.

### 3. Одна ответственность – каждый модуль делает только свою задачу (парсер адресов не знает о TLM, драйвер не знает о скриптах).

### 4. Зависимости явные – все сторонние библиотеки подключаются через find_package или FetchContent. Не допускается копирование исходников в проект.- или же включить заголовки

### 5. Кроссплатформенная обёртка для драйверов – использование условной компиляции (#ifdef _WIN32). Для Unix в будущем – отдельная реализация (например, через libusb для E20-10, если найдем). - очень на потом

### 6. Конфигурация тестов – Lua-скрипты должны иметь доступ к API для управления имитатором, вольтметром, источником питания. Никаких хардкодных IP-адресов в коде библиотеки.

### 7. Запрет на добавление новых файлов без изменения CMakeLists.txt – каждый новый .cpp/.h должен быть явно добавлен в соответствующий add_library или add_executable. Это предотвращает «случайные» включения.

### 8. Логирование – использовать внутренний логгер с уровнями (debug, info, error). Встраиваемая система может подменить вывод своим callback-ом.

### 9. Потокобезопасность – все публичные функции должны быть безопасны для вызова из разных потоков (использовать мьютексы внутри). Внутренние потоки захвата не конфликтуют с внешними вызовами.

### 10. Обработка ошибок – функции возвращают коды ошибок (int) или bool.

## 📜 Правила добавления новых файлов
### Каждый новый .h/.cpp должен быть осмысленным и соответствовать модулю.

### Перед добавлением необходимо обновить CMakeLists.txt в соответствующей директории.

### Запрещено создавать файлы с именами temp.cpp, test2.cpp – только финальные имена.

### Каждый публичный заголовок должен иметь #pragma once и быть документирован (Doxygen).

### Новые зависимости должны быть добавлены через FetchContent или find_package, с указанием лицензии.
