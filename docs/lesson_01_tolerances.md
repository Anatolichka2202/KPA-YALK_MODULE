## 1. Слоистый резолвер — это паттерн, а не «куча if-ов»

Файл: `desktop_orbita/tolerance_resolver.cpp`, метод `resolve()`.

```cpp
chstatus::Tolerance ToleranceResolver::resolve(const QString& address) const {
    const QString k = key(address);
    auto ito = override_.find(k);
    if (ito != override_.end()) return ito.value();   // слой 1: сессия
    auto itc = config_.find(k);
    if (itc != config_.end()) return itc.value();     // слой 2: .tol
    if (db_) return chstatus::forAddress(db_, address.toStdString()); // слой 3: БД
    return {};                                        // нет нигде → set=false
}
```

Это **chain of resolution / layered configuration** — тот же паттерн, что у CSS
(inline > id > class), у git config (local > global > system), у переменных
окружения. Запомни принцип: **самый специфичный источник побеждает, поиск
останавливается на первом попадании**.

Вопросы, на которые ты обязан уметь ответить:
- Почему `resolve()` помечен `const`? (Подсказка: он ничего не меняет в объекте.
  Что будет, если убрать `const` — где это «выстрелит» у вызывающего кода?)
- `override_.find(k)` возвращает итератор, а не значение. Почему не
  `override_.value(k)` или `override_[k]`? Что делает `operator[]` у `QHash`,
  если ключа НЕТ? (Это классическая ловушка — `operator[]` **вставит** пустой
  элемент. Молчаливая порча данных.)
- Почему ключ прогоняется через `key()` (нормализацию) на **обеих** сторонах —
  и при загрузке, и при поиске? Что сломается, если нормализовать только при
  загрузке?

## 2. Нормализация ключа — почему отдельная функция

```cpp
QString ToleranceResolver::key(const QString& address) {
    return QString::fromStdString(encoding::normalizeAddress(address.toStdString()));
}
```

Адрес `M16П2A40...` может прийти с кириллической `П` (из cp1251-конфига), а в
снапшоте канала будет латинская `P`. Если ключи не свести к одному виду — `QHash`
их не найдёт, потому что `"П2"` и `"P2"` для машины **разные строки**.

Правило, которое уноси с собой: **если что-то используется как ключ словаря,
канонизируй его в ОДНОМ месте и зови это отовсюду.** Дублировать
`.toUpper().replace(...)` по вызовам — гарантированный баг через месяц.

## 3. Внедрение зависимости через сеттер + фолбэк

Файлы: любой потребитель, напр. `desktop_orbita/readout_widget.cpp`.

```cpp
chstatus::Tolerance tol = (m_resolver
    ? m_resolver->resolve(QString::fromStdString(m_spec.address))
    : chstatus::forAddress(m_db, m_spec.address));   // фолбэк
```

Виджет не создаёт резолвер сам — ему его **дают** снаружи (`setToleranceResolver`).
Это dependency injection. Зачем так:
- виджет тестируемый и переиспользуемый (можно подсунуть другой резолвер);
- виджет не знает, откуда берутся допуски — это не его забота.

Тернарник с фолбэком — это **защита на время сборки системы**: пока кто-то
наверху не вызвал `setToleranceResolver`, `m_resolver == nullptr`, и код не
падает, а откатывается на старое поведение. Так делают рефакторинг по частям,
не ломая сборку на каждом шаге.

Вопрос: а это вообще правильно — навсегда оставлять фолбэк? Или это «строительные
леса», которые надо снести, когда проводка завершена? (Мой ответ: леса. Но
снести можно только убедившись, что резолвер ВСЕГДА проставлен. Найди в
`mainwindow.cpp`, где это происходит, и докажи себе, что других путей нет.)

## 4. `std::function` как граница между UI и логикой

Файл: `desktop_orbita/scenario_wizard.h`.

```cpp
using ValueProvider = std::function<std::optional<double>(const QString& addr)>;
explicit ScenarioWizard(ValueProvider provider, QWidget* parent = nullptr);
```

И в `mainwindow.cpp`:

```cpp
ValueProvider provider = [this](const QString& addr) -> std::optional<double> {
    return orbita_->getValueByAddress(addr.toStdString());
};
ScenarioWizard wizard(provider, this);
```

Смотри внимательно: `ScenarioWizard` **не знает про класс `orbita::Orbita`**. Он
знает только «есть функция, которая по адресу вернёт число или ничего». Это и есть
развязка слоёв. Если завтра данные поедут не из ядра, а из сети или из файла —
`ScenarioWizard` не меняется ни на строку, меняется только лямбда.

Это прямое следствие контракта проекта: «ядро не знает про Qt, UI не лезет в
кишки ядра». `std::function` + `std::optional` — твой инструмент для таких границ.

Вопрос: почему `std::optional<double>`, а не просто `double` с «магическим»
значением -1 для «нет данных»? (Подумай, что будет, если -1 — валидный код АЦП.)

## 5. Кроссплатформенность через `#ifdef` — и почему это костыль

Файл: `orbita/device/e2010_device.h`.

```cpp
#ifdef _WIN32
#include <windows.h>
#include "Lusbapi.h"
#else
typedef void* HANDLE;
struct OVERLAPPED { char dummy[64]; };   // ← заглушка
...
#endif
```

Это позволило собирать проект на Linux (и в облаке гонять тесты), но честно
скажу: **это технический долг**, а не архитектура. Правильно — спрятать E2010 за
интерфейсом `DeviceInterface` (он в проекте есть, но мёртв), и на Linux просто не
создавать E2010, а не подсовывать фейковые типы с `char dummy[64]`.

Задача со звёздочкой ниже про это. Пока запомни разницу: **`#ifdef` размазывает
платформенную развилку по всему файлу; интерфейс — собирает её в одной точке.**

## 6. Класс молчаливых багов: сдвиг индексов SQL

Файл: `desktop_orbita/metadata_service.cpp`, `loadMetadata()`.

Когда в `SELECT` добавили `p.id` первым столбцом, **все** `q.value(N)` ниже надо
было сдвинуть на +1:

```cpp
"SELECT p.id, p.name, p.category, ..."   // id стал value(0)
info.id   = q.value(0).toInt();
info.name = q.value(1).toString();        // было value(0)
...
```

Это опаснейший тип ошибки: компилятор **молчит** (всё это `QVariant`), приложение
запускается, но `category` показывает `signal_type`, а допуск читается из чужой
колонки. Ловится только глазами или тестом.

Мораль: после правки `SELECT` — пройди по индексам пальцем сверху вниз и сверь с
порядком столбцов. Я это проверял вручную (см. мой комментарий в чате) — и ты
приучайся.
